## Phase Done Workflow

Вызывается в конце фазы, когда все задачи выполнены. Агент закрывает фазу формально.

---

### Шаг 1: Определить контекст

1. Read `conductor/tracks.md` → найди `🔵 Active` трек
2. Read `gate.md` из папки активного трека
3. Определить текущую фазу:
   - Если пользователь указал номер → эта фаза
   - Иначе → последняя фаза с `[/]` (in-progress) пунктами
   - Если непонятно → спросить пользователя

---

### Шаг 2: Аудит задач фазы

Проверить gate.md для этой фазы:

```
✅ Завершены: [список [x] задач]
🔲 Не завершены: [список [ ] задач]  
🔄 В процессе: [список [/] задач]
```

**Если есть незавершённые `[ ]` или `[/]`:**

- ⚠️ Предупредить: «N задач не завершены. Закрыть фазу с пропусками?»
- Ждать ответа пользователя
- Если пользователь говорит «да» → пометить незавершённые как `[ ]` (оставить) и добавить заметку в walkthrough
- Если «нет» → прервать, перечислить что осталось доделать

---

### Шаг 3: Gate check

Запустить проверки указанные в gate фазы:

```bash
pnpm type-check 2>&1 | tail -10
```

```bash
pnpm build 2>&1 | tail -10
```

```bash
pnpm lint 2>&1 | tail -10
```

Если в gate указан `pnpm test`:

```bash
pnpm test 2>&1 | tail -20
```

**Результат:**

- Все Exit 0 → ✅ продолжить
- Ошибки → ⚠️ показать ошибки, спросить: «Чинить или закрыть фазу с проблемами?»

---

### Шаг 4: Обновить gate.md

- Все `[/]` → `[x]` для завершённых задач
- Проверить что ничего не пропущено
- НЕ менять `[ ]` на `[x]` если задача реально не выполнена

---

### Шаг 5: Записать walkthrough

Создать/дополнить файл:

```
conductor/tracks/track-N-<slug>/walkthrough.md
```

**Если файл уже существует** → добавить новый раздел в конец (НЕ перезаписывать):

```markdown
## Phase N — [Title]
> Дата: YYYY-MM-DD · Агент: [код модели]

### Что сделано
- [конкретное изменение 1 с именем файла]
- [конкретное изменение 2]

### Файлы изменены
| Файл | Изменение |
|------|-----------|
| `path/to/file.ts` | описание |

### Верификация
- `pnpm type-check` → ✅/❌
- `pnpm build` → ✅/❌
- `pnpm test` → ✅/❌/—

### Заметки для следующего агента
- [нюанс который стоит знать]
- [решение которое было принято и почему]
```

---

### Шаг 5.5: Kaizen Review 🔍

Пока контекст максимальный — подумай что узнал в процессе выполнения этой фазы.

**Проанализировать:**

- Что оказалось сложнее/проще чем ожидалось
- Что обнаружил в коде что влияет на следующие задачи
- Где реальный код отличается от того что написано в gate.md
- Что стоит знать следующему агенту но не вошло в walkthrough «Заметки»

**Результат:**
Запиши все инсайты, риски для будущих фаз и предложения в `walkthrough.md` в раздел `### Заметки для следующего агента`. Следующий агент сам прочитает их при старте задачи.

⚠️ НЕ пытайся изменять `gate.md` следующих фаз на лету, чтобы случайно не сломать бизнес-контекст трека!

---

### Шаг 6: CHANGELOG.md

Добавить запись в начало CHANGELOG (после заголовка) в формате:

```markdown
### [YYYY-MM-DD] Track N Phase N — [Title]

**Added:**
- новая фича

**Changed:**
- изменение

**Fixed:**
- исправление
```

Если за текущую дату уже есть запись → дополнить её.

---

### Шаг 7: Очистка

- Удалить локальные планы: `rm phase-N-plan.md` (если существует).
- Если использовался скилл `planning-with-files` (Manus-style) - удалить `task_plan.md`, `findings.md` и `progress.md` из корня проекта.
- Выполни `git status --short` → показать что изменилось

---

### Шаг 8: Прогресс + доклад

Посчитать общий прогресс трека:

- Всего фаз в gate.md
- Все `[x]` vs все `[ ]` задачи
- Процент завершения

Вывести:

```
✅ Phase N — [Title] завершена

📊 Прогресс трека: Phase N из M · [Done]/[Total] задач · ~XX%
🧪 type-check: ✅ · build: ✅ · test: ✅/—
📝 walkthrough.md обновлён
📋 CHANGELOG.md обновлён

🗂 Изменённые файлы:
   [git status output]

🎯 Следующий шаг: Запусти `[/review]` перед коммитом! (1 commit = 1 task)

🎯 Следующая фаза: Phase N+1 — [Title]
   Запусти `/phase` в этом или новом чате.

💡 Рекомендация: [если фаза маленькая → "можно продолжить /phase N+1 в этом чате"
                  если большая → "лучше новый чат для свежего контекста"]
```

---

## Правила

- `/done` вызывается ТОЛЬКО после завершения работы, НЕ для планирования
- Если gate check падает → НЕ закрывать фазу автоматически, спросить пользователя
- Walkthrough ДОПОЛНЯЕТСЯ (append), никогда не перезаписывается
- CHANGELOG обязателен — без него фаза не считается закрытой
- Рекомендация «новый чат или продолжить» основана на объёме следующей фазы
