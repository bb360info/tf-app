## Agent Onboarding Workflow

Выполни ВСЕ шаги по порядку перед тем как отвечать на задачу.

## Режимы выполнения (оптимизация)

Перед шагом 0 определи режим:

- `fast` — если пользователь уже дал конкретную задачу в первом сообщении.
- `full` — если это чистый онбординг без конкретной задачи (старт дня/сессии, handoff).

### `fast` (по умолчанию при конкретной задаче)

Выполняй только:

1. Шаг 0 (самоидентификация модели)
2. Шаг 2 (активный трек из `conductor/tracks.md`)
3. Шаг 3 (только `gate.md` active track; `context.md` читать только если задача завязана на бизнес-контекст)
4. Шаг 8 (Skill Loading по реальной задаче пользователя)
5. Короткий доклад: активный трек + выбранные скиллы + что делаем дальше

Пропусти в `fast`:

- Шаг 1 (`GEMINI.md`)
- Шаг 4 (чтение всех планов)
- Шаг 5 (health check)
- Шаг 6 (`CHANGELOG.md`)
- Шаг 7 (поиск "следующей задачи из gate", если пользователь уже дал свою)
- Полный доклад из шага 9

### `full` (только когда нет конкретной задачи)

Выполняй исходный полный workflow (шаги 0-10).

### Шаг 0. Самоидентификация модели (The Iron Title Rule)

Определи, какая модель тебя выполняет, и запомни её код. Согласно **GEMINI.md (Rule 7)**, этот код ДОЛЖЕН быть в самой первой строке твоего первого ответа в чате, чтобы он отобразился в названии:

| Код | Модель |
| --- | ------ |
| `G1H` | Gemini 3.1 Pro (High) |
| `G1L` | Gemini 3.1 Pro (Low) |
| `G3H` | Gemini 3 Pro (High) |
| `G3L` | Gemini 3 Pro (Low) |
| `GF`  | Gemini 3 Flash |
| `CS`  | Claude Sonnet 4.6 (Thinking) |
| `CO`  | Claude Opus 4.6 (Thinking) |
| `??`  | Неизвестная модель |

Код модели будет использован в первой строке доклада.

### Шаг 1. Правила проекта

Read `GEMINI.md` — правила проекта, архитектура, дизайн-система, execution rules.

### Шаг 2. Активный трек

Read `conductor/tracks.md`:

- Ищи строку `🔵 Active` → запомни номер трека и название
- Если `🔵 Active` **нет** → ищи `🟡 Next` → предупреди: «Нет активного трека. Следующий: Track N»
- Предложи: «Запустить `/phase` для активации?»

### Шаг 3. Gate + Context активного трека

Автоматически определи путь к папке активного трека:

- Из строки `🔵 Active | N | Title` → папка `conductor/tracks/track-N-<slug>/`
- Например: `🔵 Active | 3 | Training System` → папка `track-3-training/`

Читай **оба файла** (или используй light-режим, см. ниже):

1. `gate.md` — отметь что ✅ сделано, что `[ ]` осталось
2. `context.md` — прочитай контекст задачи (что делаем, зачем, какие файлы затрагивает)

#### Light-режим для повторного `/switch-agent` в той же сессии

1. Проверь `conductor/tracks/track-N-<slug>/workflow_state.md`.
2. Если есть блок `last_switch_agent_context` и hashes `gate.md`/`context.md` совпадают с текущими:
   - Не перечитывай файлы полностью.
   - Прочитай только summary из `workflow_state.md`.
3. Если hashes отличаются или state отсутствует — выполняй full-read.

### Шаг 4. Планы предыдущего агента

Проверь наличие планов в папке трека (читай **в порядке приоритета**):

1. Если **`phase-N-plan.md` существует** — ОБЯЗАТЕЛЬНО прочитай (предыдущий агент начал фазу, нужно продолжить)
2. Если **`implementation_plan.md` существует** — прочитай (план трека от предыдущего агента)
3. Если **`walkthrough.md` существует** — читай ТОЛЬКО если нужен контекст предыдущей работы (пропусти если свежий plan уже есть)

> 💡 Оптимизация: не читай walkthrough если phase-N-plan.md или implementation_plan.md уже содержат достаточно контекста.

### Шаг 5. Health check (опциональный)

Запускай ТОЛЬКО если пользователь **не дал** конкретную задачу И это начало нового рабочего дня/сессии:

```bash
pnpm type-check 2>&1 | tail -5
```

Если есть ошибки TS, первая строка ТВОЕГО доклада должна быть:
`[КОД_МОДЕЛИ] ❌ ВНИМАНИЕ: Проект не собирается (TS errors). Сначала чиним билд!`

> ⚡ Skip type-check если: пользователь дал явную задачу, /phase был только что завершён, или последний gate check показал ✅.

### Шаг 6. CHANGELOG

Read `CHANGELOG.md` (последние 20-30 строк) — изучи последние изменения в коде:

```bash
tail -30 CHANGELOG.md
```

### Шаг 7. Следующая задача

Найди следующую незакрытую задачу в gate.md:

- Первый `[ ]` пункт = ближайшая задача
- Если есть незакрытые зависимости (deferred) — учти их

### Шаг 8. Загрузка скиллов

1. Read `.agent/skills/project_skills.json` — вспомни доступные группы скиллов и `_blocklist`
2. Определи какие группы скиллов релевантны для **следующей задачи** (из шага 7)
3. **Загрузи `SKILL.md`** для каждой выбранной группы — используй `view_file` на каждый `.agent/skills/skills/<group>/SKILL.md`

> [!IMPORTANT]
> Это обязательно (Skill Loading Iron Law из GEMINI.md). Лимит по умолчанию: **2-4 скилла**. Расширяй до 5-7 только для high-risk задач. Проверь `_blocklist` — заблокированные скиллы НИКОГДА не грузить.

### Шаг 9. Доклад

Сделай краткий доклад в формате:

```text
[КОД_МОДЕЛИ] 🔵 Активный трек: Track N — [Название]

✅ Готово (N из M):
- Список завершённых пунктов

🆕 Последние изменения (из CHANGELOG.md):
- краткое описание последних 3-5 изменений

📐 План от предыдущего агента: ✅ Найден / ❌ Нет
📝 Walkthrough предыдущей работы: ✅ Найден / ❌ Нет

🎯 Следующая задача: [первый незакрытый пункт]

📋 Осталось: N пунктов
- [ ] пункт 1
- [ ] пункт 2
...

🛠 Скиллы загружены: [список прочитанных SKILL.md]
```

Первая строка доклада ВСЕГДА начинается с кода модели в квадратных скобках, например:
`[CS] 🔵 Активный трек: Track 5 — Video + Biomechanics`

### Шаг 10. Рекомендация «что дальше»

- Если найден `phase-N-plan.md` → «Фаза N спланирована. Продолжить выполнение?»
- Если найден `implementation_plan.md` → «Найден план трека. Запустить `/phase` для следующей фазы?»
- Иначе → «Начинаем с [следующая задача]? Запусти `/phase` для kickoff.»
- Если пользователь уже дал конкретную задачу → выполняй сразу

## Правила сохранения артефактов между агентами

**Автоматически (через `/phase`):**

- `phase-N-plan.md` → создаётся при утверждении плана (шаг 7.5)
- `walkthrough.md` → обновляется при завершении фазы (шаг 8c)
- Удаляется `phase-N-plan.md` после завершения (шаг 8e)

**Для агента-планировщика (Opus):** Если создаёт track-level implementation_plan.md в brain artifacts:

```bash
cp <brain-artifact>/implementation_plan.md → conductor/tracks/track-N-<slug>/implementation_plan.md
```

Это позволяет следующему агенту автоматически подхватить контекст через `/switch-agent`.

## Обновление `workflow_state.md` после доклада

Обнови в active track:

- `last_read_docs_versions.gate_md`
- `last_read_docs_versions.context_md`
- `last_read_docs_versions.changelog_tail`
- `last_loaded_skills`
- `last_switch_agent_context` (короткое summary + mode full/light)
