## Agent Onboarding Workflow

Выполни ВСЕ шаги по порядку перед тем как отвечать на задачу.

0. **Самоидентификация модели** — определи, какая модель тебя выполняет, и запомни её код:

   | Код | Модель |
   |-----|--------|
   | `G1H` | Gemini 3.1 Pro (High) |
   | `G1L` | Gemini 3.1 Pro (Low) |
   | `G3H` | Gemini 3 Pro (High) |
   | `G3L` | Gemini 3 Pro (Low) |
   | `GF` | Gemini 3 Flash |
   | `CS` | Claude Sonnet 4.6 (Thinking) |
   | `CO` | Claude Opus 4.6 (Thinking) |
   | `??` | Неизвестная модель |

   Код модели будет использован в первой строке доклада.

1. Read `GEMINI.md` — правила проекта, архитектура, дизайн-система, execution rules

2. Read `conductor/tracks.md` — найди строку со статусом **🔵 Active**, запомни номер трека и название

3. Автоматически определи путь к gate активного трека:
   - Из строки `🔵 Active | N | Title` → папка называется `track-N-<slug>`
   - Например: `🔵 Active | 3 | Training System` → читай `conductor/tracks/track-3-training/gate.md`
   - Прочитай gate.md: отметь что ✅ сделано, что `[ ]` осталось

4. **Проверь наличие планов и walkthroughs в папке трека:**
   - Проверь файл `conductor/tracks/track-N-<slug>/implementation_plan.md`
   - Проверь файл `conductor/tracks/track-N-<slug>/walkthrough.md`
   - Если **implementation_plan.md существует** — прочитай его, это план от предыдущего агента
   - Если **walkthrough.md существует** — прочитай его, это отчёт о предыдущей работе

4.5. **Health check** — запусти обязательную проверку состояния проекта ПЕРЕД докладом (не делай этого если пользователь уже дал четкую задачу и она не требует проверки!):
   ```bash
   pnpm type-check 2>&1 | tail -5
   ```
   Если есть ошибки TS, первая строка ТВОЕГО доклада должна быть:
   `[КОД_МОДЕЛИ] ❌ ВНИМАНИЕ: Проект не собирается (TS errors). Сначала чиним билд!`

5. Read `CHANGELOG.md` (весь файл) — изучи последние изменения в коде. Это даёт контекст что реально было сделано недавно.

6. Найди следующую незакрытую задачу в gate.md:
   - Первый `[ ]` пункт = ближайшая задача
   - Если есть незакрытые зависимости (deferred) — учти их

7. Read `.agent/skills/project_skills.json` — вспомни доступные группы скиллов и blocklist

8. Сделай краткий доклад в формате:
   ```
   [КОД_МОДЕЛИ] 🔵 Активный трек: Track N — [Название]

   ✅ Готово (N из M):
   - Список завершённых пунктов

   🆕 Последние изменения (из CHANGELOG.md):
   - краткое описание последних 3-5 изменений

   📐 План от предыдущего агента: ✅ Найден / ❌ Нет
   📝 Walkthrough предыдущей работы: ✅ Найден / ❌ Нет

   🎯 Следующая задача: [первый незакрытый пункт]

   📋 Осталось: N пунктов
   - [ ] пункт 1
   - [ ] пункт 2
   ...

   🛠 Скиллы для этой задачи: [группы из project_skills.json]
   ```

   Первая строка доклада ВСЕГДА начинается с кода модели в квадратных скобках, например:
   `[GT] 🔵 Активный трек: Track 5 — Video + Biomechanics`

9. Если найден implementation_plan.md — спроси: "Найден план. Начать выполнение?"
   Иначе — спроси: "Начинаем с [следующая задача]?" или выполни если пользователь уже дал задачу

## Правила сохранения артефактов между агентами

**Для агента-планировщика (Opus):** Когда пользователь утверждает implementation_plan.md, скопируй его в папку активного трека:
```
cp <brain-artifact>/implementation_plan.md → conductor/tracks/track-N-<slug>/implementation_plan.md
```

**Для агента-исполнителя (Sonnet):** После завершения работы, сохрани walkthrough.md в папку трека:
```
cp <brain-artifact>/walkthrough.md → conductor/tracks/track-N-<slug>/walkthrough.md
```

Это позволяет следующему агенту автоматически подхватить контекст через `/switch-agent`.
