## Phase Kickoff Workflow

Выполни ВСЕ шаги по порядку. Цель — создать детальный план для следующей незакрытой фазы из gate.md, проверить зависимости и конфликты, и показать план пользователю перед исполнением.

> ⚠️ **Правило:** Нельзя написать план (Шаг 6) пока не закрыты все чекбоксы в Шаге 4. «Знаю этот скилл» ≠ «прочитал SKILL.md».

---

### Шаг 1: Найти активный трек + определить фазу

1. Read `conductor/tracks.md`:
   - Ищи `🔵 Active` → запомни N и slug
   - Если `🔵 Active` **нет** → ищи `🟡 Next` → это трек, который будем начинать
   - Если ни того ни другого → ⚠️ спросить пользователя какой трек запускать
2. Построй путь: `conductor/tracks/track-N-<slug>/`
3. Read `gate.md` и `context.md` из этой папки
4. Найди `phase-N-plan.md` для целевой фазы (если пользователь не указал фазу, сначала определи её по gate.md).
5. Если `phase-N-plan.md` существует и **decision-ready** (есть `## Phase N`, `### Задача`, `Verify:`) и файл свежий (обновлён <= 24h назад):
   - Используй `light`-путь: **не читай `implementation_plan.md` повторно**
   - Прочитай только `phase-N-plan.md` + `gate.md` + `context.md`
6. Если `phase-N-plan.md` отсутствует/неполный/устарел:
   - Read `implementation_plan.md` (если есть)
   - Если `implementation_plan.md` нет → read `walkthrough.md` последние 100 строк только:

   ```bash
   tail -100 conductor/tracks/track-N-<slug>/walkthrough.md
   ```

   > 💡 Не читай walkthrough если implementation_plan.md уже есть — он содержит достаточно контекста.
7. Read `CHANGELOG.md` (последние ~30 строк):

   ```bash
   tail -30 CHANGELOG.md
   ```

**Автосмена статуса:**

- Если трек был `🟡 Next` → обновить `conductor/tracks.md`: заменить `🟡 Next` → `🔵 Active`
- Предупредить пользователя: «Трек N переведён в 🔵 Active»

**Определение фазы:**

- Если пользователь указал номер → взять указанную фазу
- Если нет → первая фаза с хотя бы одним `[ ]` пунктом
- Убедись что ВСЕ предыдущие фазы закрыты (`[x]`)
- Если предыдущая не закрыта → ⚠️ предупредить, уточнить у пользователя

Извлечь из gate.md:

- Заголовок и оценку времени
- Строку `**Skills:**` для этой фазы (если есть на уровне фазы)
- Строку `**Skills (N):**` из шапки gate (общие скиллы трека)
- Наличие `**Mandatory reads:**`
- Все `[ ]` задачи фазы

---

### Шаг 2: Прогресс трека

Посчитай общий прогресс gate.md:

- Всего `[ ]` + `[x]` + `[/]` = Total tasks
- `[x]` = Done tasks
- Текущая фаза = N из M фаз

Отобразить в итоговом отчёте:

```
📊 Phase N из M · [Done]/[Total] задач · ~XX% завершено
```

---

### Шаг 3: Health check (условный)

> ⚡ **Пропусти** если: `/switch-agent` уже запускался в этом чате, или пользователь дал явную задачу без запроса проверки состояния.

```bash
pnpm type-check 2>&1 | tail -5
```

- Exit 0 → ✅
- Ошибки → ⚠️ показать — решить с пользователем, чинить ли сначала

```bash
git status --short
```

- Clean → ✅
- Uncommitted → ⚠️ предупредить (риск конфликтов при merge)

---

### Шаг 4: Загрузить скиллы

Теперь, когда известна задача (из Шагов 1-3), грузить скиллы целенаправленно:

**4a. Проверить blocklist:**

Read `.agent/skills/project_skills.json` → запомни `_blocklist`. Заблокированные скиллы НИКОГДА не грузить.

**4b. Execution скиллы** — определяются по типу работы в фазе:

- Если фаза указывает `**Skills:**` → грузить перечисленные
- Если нет → выбрать из групп в `project_skills.json` по типу задачи:
  - Backend/service → `typescript-expert`, `code-refactoring-refactor-clean`
  - UI/компоненты → `react-patterns`, `react-ui-patterns`, `jumpedia-design-system`
  - CSS/визуал → выполнить `/ui-work` workflow (загружает design-system + токены)
  - Баги → `systematic-debugging`
  - Рефакторинг → `code-refactoring-refactor-clean`, `architect-review`

> ⚠️ **UI-задачи:** `jumpedia-design-system` и `verification-before-completion` НЕ грузятся вручную.
>
> - `jumpedia-design-system` → загружается через `/ui-work` (только если фаза UI/CSS)
> - `verification-before-completion` → загружается на Шаге 7 (только перед финальной верификацией)

**4c. Планирование скиллы** — для написания плана:

- **Всегда**: `concise-planning` — структура плана, атомарные задачи
- **Если фаза большая** (5+ задач) → добавить `brainstorming`
- **Если архитектурные решения** → добавить `context-driven-development`

**4d. Загрузка:**

Загрузить `SKILL.md` для каждого выбранного скилла через `view_file`:
`.agent/skills/skills/<id>/SKILL.md`

- **Бюджет по умолчанию: 2-4 SKILL.md на задачу**
- **High-risk исключение:** до 5 SKILL.md (security/auth/DB migrations/critical refactor)
- Если в Skills или gate есть `/ui-work` → выполнить ПОЛНЫЙ `/ui-work` workflow
- Если `mandatory_reads` указаны → прочитать эти файлы

**⛔ Чеклист загрузки — ОБЯЗАТЕЛЬНО закрыть перед шагом 6:**

```
- [ ] [скилл 1] SKILL.md прочитан через view_file
- [ ] [скилл 2] SKILL.md прочитан через view_file
- [ ] concise-planning SKILL.md прочитан через view_file
...
Если хотя бы один не прочитан → прочитай сейчас. НЕЛЬЗЯ писать план дальше.
```

> ⚠️ «Знаю этот скилл по памяти» ≠ «прочитал SKILL.md». Всегда читать через `view_file`.

---

### Шаг 4.5: Маркировка прогресса в gate.md

- При **начале** работы над фазой → пометить все `[ ]` задачи фазы → `[/]`
- Это позволяет следующему агенту видеть что было начато но не завершено
- При **завершении** каждой задачи → `[/]` → `[x]`

---

### Шаг 5: Изучить кодовую базу для фазы

Это ключевой шаг — **агент ОБЯЗАН прочитать все файлы**, которые будут затронуты в этой фазе, перед написанием плана:

1. Из `context.md` + `implementation_plan.md` определить список файлов для этой фазы
2. Прочитать каждый существующий файл, который будет изменён
3. Если фаза создаёт новые файлы → прочитать аналоги (похожие сервисы, компоненты) как референс
4. Если фаза работает с PB → через MCP проверить нужные коллекции и поля
5. Если фаза зависит от предыдущей (новые типы, сервисы) → проверить что файлы существуют + прочитать экспортируемые типы/функции

`Light`-оптимизация:

- Если на Шаге 1 найден свежий `phase-N-plan.md`, не перечитывай повторно уже зафиксированные в нём файлы без причины.
- Дочитывай только изменившиеся или конфликтные участки.

---

### Шаг 6: Написать детальный план фазы

На основе изученной кодовой базы создать **детальный план** — конкретный, не абстрактный:

**Структура плана для каждой задачи из gate.md:**

```
## Phase N — [Title]

### Задача: [текст из gate.md]
**Файл:** `path/to/file.ts`
**Что делать:**
- Добавить функцию `foo(bar: string): Promise<Baz>` после `existingFn()`
- Изменить тип аргумента `x` с `string` на `AthleteId` (branded type)
- Убрать raw string interpolation в строке 42 → заменить на `pb.filter('id = {:id}', { id })`
**Verify:** [конкретная команда или проверка из gate.md]

### Задача: [следующая]
...
```

Требования к плану:

- **Конкретно**: называй функции, типы, пропсы, CSS-классы — без «добавить компонент»
- **Порядок**: если задача A нужна для задачи B → задача A идёт первой
- **Конфликты**: если два изменения правят один файл — объединить в одну задачу
- **Зависимости**: явно указать если задача зависит от PB schema, другого файла или предыдущей задачи
- **Риски**: отметить нестандартные решения или потенциальные проблемы

---

### Шаг 7: Показать план пользователю

**⛔ Самопроверка перед показом плана:**

```
- [ ] Все SKILL.md из шага 4d реально прочитаны (не просто задекларированы)
- [ ] Верификация включает pnpm lint (если JS/TS проект)
- [ ] Риски circular imports при re-export отмечены (если есть re-export)
- [ ] Порядок задач учитывает зависимости (A нужен для B → A первый)
- [ ] context.md прочитан и учтён в плане
```

Если хотя бы один не закрыт → дозагрузи скиллы / добавь в план.

> **Перед финальной верификацией** (когда код написан) загрузи:
> `view_file .agent/skills/skills/verification-before-completion/SKILL.md`

Вывести:

```
🟢 Phase N — [Title] (~N часов)

📊 Phase N из M · [Done]/[Total] задач · ~XX% трека завершено

📋 Задачи: X пунктов из gate.md

🛠 Скиллы загружены: [список прочитанных SKILL.md]

⚙️ Зависимости:
- Предыдущие фазы: ✅ / ⚠️
- TypeScript: ✅ / ⚠️ N ошибок
- Git: ✅ / ⚠️ Uncommitted
- PB Schema: ✅ / ⚠️ (если применимо)
- Context: ✅ прочитан
- Walkthrough: ✅ Найден / — Нет

━━━━━━━━━━━━━━━━━━━━━━━━
📐 ДЕТАЛЬНЫЙ ПЛАН
━━━━━━━━━━━━━━━━━━━━━━━━
[план из Шага 6]

━━━━━━━━━━━━━━━━━━━━━━━━
Подтверди план — начну исполнение.
```

**Ждать подтверждения от пользователя перед началом исполнения.**
Если пользователь говорит «да / ок / поехали / start» → переходить к исполнению.
Если просит правки → скорректировать план и показать снова.

---

### Шаг 7.5: Сохранить план в файл

После утверждения плана пользователем — сохранить детальный план в папку трека:

```
conductor/tracks/track-N-<slug>/phase-N-plan.md
```

Если файл уже существует → предупредить: «`phase-N-plan.md` уже существует — перезаписать?» и дождаться ответа.

Формат файла:

```markdown
# Phase N — [Title]
> Дата: YYYY-MM-DD · Агент: [код модели] · Скиллы: [список]

[детальный план из Шага 6]
```

Зачем: следующий агент при `/switch-agent` или `/phase` увидит что было запланировано и сможет продолжить, даже если чат потерян.

---

### Шаг 8: Завершение фазы → `/done`

Перед вызовом `/done` убедись что:

```
- [ ] Все задачи фазы в gate.md помечены [x]
- [ ] Нет ни одного [/] (в прогрессе) в текущей фазе
- [ ] pnpm type-check прошёл без ошибок
- [ ] pnpm build прошёл (или пользователь дал добро пропустить)
```

Только после закрытия всех чекбоксов → вызвать `/done`:

- Gate check (type-check + build + test)
- Обновление gate.md (`[/]` → `[x]`)
- Walkthrough append
- CHANGELOG update
- Очистка phase-N-plan.md
- Прогресс + доклад

> **Важно:** `/done` — отдельный workflow. Не пытайся делать завершение вручную.
> Подробности: `.agent/workflows/done.md`

---

## Важно

- `/phase` НЕ заменяет `/switch-agent` — они комплементарны
  - `/switch-agent` = онбоардинг в начале чата (полный обзор трека, модель, CHANGELOG)
  - `/phase` = kickoff конкретной фазы (глубокое изучение + план + старт)
- `/phase` можно вызывать ПОСЛЕ `/switch-agent` или самостоятельно (если контекст уже известен)
- НИКОГДА не грузить скиллы из `_blocklist`
- Если context.md или implementation_plan.md отсутствуют → изучить gate.md + прочитать файлы напрямую
- Автосмена `🟡 Next` → `🔵 Active` в tracks.md — обязательна при первом запуске фазы нового трека

## Артефакт состояния (`workflow_state.md`)

Для active track поддерживай файл:
`conductor/tracks/track-N-<slug>/workflow_state.md`

Минимум полей для `/phase`:

- `last_loaded_skills`
- `last_read_docs_versions.project_skills_json`
- `last_phase_context.phase`
- `last_phase_context.used_mode` (`full` / `light`)
- `last_phase_context.plan_file`
