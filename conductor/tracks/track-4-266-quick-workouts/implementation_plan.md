# Быстрые тренировки: Контекстно-зависимый гибрид (Вариант С3)

Этот документ описывает технический план реализации для функции быстрых тренировок с обработкой конфликтов при создании сезона (Вариант C3).

## Цель

Позволить тренерам и атлетам назначать "быстрые тренировки" на любые даты, даже если на эти даты нет активного сезона. При последующем создании сезона, перекрывающего эти тренировки, система должна предложить тренеру варианты разрешения конфликтов (объединить, оставить как есть, удалить).

## Предлагаемые изменения

### Backend (PocketBase Services)

#### [MODIFY] `src/lib/pocketbase/services/plans.ts`

* **Изменение `getOrCreatePlan`:** Эта функция сейчас возвращает ошибку, если сезон/фаза не найдены, так как она полагается на наличие `phaseId`. Мы создадим новую функцию `ensureStandalonePlan` специально для создания планов типа `standalone`. В `training_plans` `phase_id` уже опционален, поэтому мы можем создавать планы без него.
* **Новая функция `getStandalonePlansForPeriod(athleteId, startDate, endDate)`:** Будет возвращать все планы с `plan_type = 'standalone'` для данного атлета в заданном диапазоне дат. Это необходимо для обнаружения конфликтов при создании сезона.
* **Новая функция `resolveStandaloneConflict(planId, action, targetPhaseId)`:** Обработает выбор тренера:
  * `action === 'merge'`: Меняет `plan_type` на `override` и присваивает `phase_id` = `targetPhaseId`.
  * `action === 'keep'`: Оставляет план как `standalone`, не привязывая к фазе.
  * `action === 'delete'`: Помечает план как удаленный (`deleted_at`).

#### [MODIFY] `src/lib/pocketbase/services/seasons.ts`

* **Обновление логики `createPhasesForSeason` (опционально, или вызов из `SeasonWizard`):** Логика разрешения конфликтов будет в основном управляться с фронтенда (Wizard вызовет проверки и решения), но сервисы должны быть готовы предоставить данные о существующих "standalone" планах.

### Frontend (User Interface)

#### [MODIFY] `src/components/training/QuickWorkout.tsx`

* **Изменение логики сохранения (`ensurePlanAndAssign`):**
  * Убрать жесткое требование (`throw new Error(t('noSeasonForDate'))`), если сезон не найден.
  * Если сезон найден — работаем по старому сценарию (добавляем тренировку в план фазы или создаем `override`).
  * Если сезон НЕ найден — создаем новый `standalone` план на эту дату (используя новую функцию `ensureStandalonePlan`) и добавляем упражнения туда.

#### [MODIFY] `src/components/training/SeasonWizard.tsx` (или новый компонент `SeasonConflicts.tsx`)

* Мы добавим новый шаг в визард создания сезона (между `competitions` и `review`, или внутри `phases`) для разрешения конфликтов.
* **Логика:**
    1. Перед переходом к созданию (или на финальном шаге) запрашиваем `getStandalonePlansForPeriod`.
    2. Если найдены `standalone` планы, показываем UI разрешения конфликтов: "Найдены независимые тренировки в датах нового сезона. Что с ними сделать?".
    3. Для каждой даты с конфликтом (или оптом) предлагаем выбор:
        * **Включить в сезон (Merge):** Тренировка станет перекрытием (`override`) для соответствующей фазы.
        * **Оставить независимо (Keep):** Системы будут существовать параллельно (например, базовая тренировка утром, быстрая вечером).
        * **Удалить (Delete):** Старая тренировка будет удалена.
* **Интернационализация:** Добавление ключей перевода для нового UI в `ru.json`, `en.json`, `cn.json`.

## План проверки (Verification Plan)

### Automated Tests

1. **TypeScript Compilation:** `pnpm type-check` (убедиться, что новые функции сервисов и изменения в компонентах типизированы верно).
2. **Linting:** `pnpm lint` (проверка стандартов кода).
3. **Build Validation:** `pnpm build` (убедиться, что Static Export работает без ошибок после изменений).

### Manual Verification

1. **Создание Standalone тренировки:**
    * Зайти под тренером, открыть Dashboard.
    * Выбрать атлета, у которого нет активного сезона на выбранную дату (например, на следующий месяц).
    * Нажать "Quick Workout", добавить пару упражнений, нажать "Assign".
    * Убедиться, что сохранение проходит успешно (нет ошибки "noSeasonForDate").
    * Посмотреть базу данных (в PocketBase Admin) — должен появиться `training_plans` с `plan_type='standalone'` и `phase_id=null`.
2. **Обработка конфликта при создании сезона (Merge):**
    * Начать создание нового сезона в `SeasonWizard`, охватывающего дату только что созданной standalone тренировки.
    * Дойти до проверки конфликтов. Должен появиться интерфейс, показывающий найденную тренировку.
    * Выбрать действие "Merge" (Включить в сезон) и завершить создание сезона.
    * Проверить в БД: у этого `training_plans` поле `plan_type` должно измениться на `override`, а `phase_id` установиться на ID соответствующей новой фазы сезона.
3. **Обработка конфликта (Keep/Delete):**
    * Повторить шаг 1 (создать новую standalone).
    * Создать сезон. Выбрать "Delete". Завершить. Убедиться, что план удален (soft-delete).
    * Повторить шаг 1. Создать сезон. Выбрать "Keep". Завершить. Убедиться, что план остался `standalone` и `phase_id=null`.
