# Контекст: Track 4.243 — Competitions Hub + History + Auto-PR

## Почему появился этот трек

После обсуждения UX и data-flow было принято решение вынести соревнования в отдельный продуктовый контур:
- не смешивать историю стартов с тренировочным планом;
- сделать единый экран истории и предстоящих соревнований;
- упростить логику PR: убрать ручной ввод и считать PB из официальных результатов.

Трек 4.243 создаётся как мост между polish-треком 4.242 и heavy-tech треком 5 (Video + Biomechanics).

## Проблемы текущего состояния

1. История прошлых стартов не имеет единого UI потока.
2. Логика PR дублирует фактические результаты (`personal_records` vs competition facts).
3. Нет role-safe механизма, когда атлет предлагает правку, а тренер принимает финальное решение.
4. Нет централизованного обзора "все результаты всех атлетов" для тренера по сезонам/стартам.

## Целевое состояние

1. Отдельная страница `/competitions`:
- вкладка `Upcoming`;
- вкладка `History`;
- единый формат добавления стартов и правок.

2. Двухконтурная модель данных:
- `official` слой (финальные данные тренера);
- `proposal` слой (предложения атлета с audit trail).

3. PR как производная метрика:
- берётся из `official_result`;
- фильтруется по дисциплине и season_type (`indoor`/`outdoor`);
- отображается с контекстом "где и когда установлен".

4. Мульти-атлетный реестр для тренера:
- старт + список всех участников + статусы/результаты/pending proposals.

## Ролевая модель (зафиксировано)

### Coach
- создаёт/редактирует старт;
- назначает участников;
- вносит official результат напрямую;
- approve/reject proposals (с возможностью правки);
- модерирует media (hide/unhide).

### Athlete
- видит свои upcoming/history старты;
- предлагает изменения метаданных/результатов;
- добавляет pre-event информацию и media;
- не может напрямую менять official поля.

## UX-принципы

1. Training = "что делать сейчас"; Competitions = "история и старты".
2. Соревнование всегда кликабельно в карточку старта.
3. Везде, где есть потенциальный конфликт данных, показывается явный статус (`pending`, `approved`, `rejected`).
4. У coach и athlete одна и та же карточка старта, но разные actions.

## Технические ограничения и риски

1. Нужна migration/архивация legacy `personal_records` перед отключением runtime-зависимости.
2. Нужна строгая настройка PB rules для предотвращения незаметного overwrite official данных.
3. Возможны race conditions при массовом approve proposals в одном старте (нужно serial/optimistic control).
4. Для производительности потребуется индексирование и/или projection view под PR queries.

## Критерии успешности трека

1. Атлет может заполнить историю прошлых соревнований через proposal flow.
2. Тренер видит все результаты команды и управляет конфликтами в одном реестре.
3. PR в интерфейсе вычисляется без ручного ввода и совпадает с official competition data.
4. Training page не перегружен историей и остаётся focused на дневном плане.
